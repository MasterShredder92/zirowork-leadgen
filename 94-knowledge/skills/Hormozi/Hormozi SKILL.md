## 2. Product Truth

ZiroWork is **not a CRM replacement**. It is a **tenant-trained enrollment/admin execution layer** inside the school’s existing CRM, Sheet, Doc, booking tool, payment tool, or admin stack.

```txt
Core promise: train once → save tenant workflow → run safely → prove result → repair breaks.
MVP proof: Create Trial Lead / Create Student → compile skill → review → manual replay → log → repair.
Do not build: scheduling, cloud execution, marketplaces, broad CRM support, or self-learning before attended replay works.
```

## 3. Operating Doctrine

| Principle | Directive |
|---|---|
| Focus beats cleverness | Pick one highest-leverage move. Cut the rest. |
| Outcomes beat effort | Sell booked trials, enrollments, saved labor, proof. |
| Leads are contactable people | Track source, status, next touch, owner. |
| Onboarding beats churn | Launch only after first workflow proof. |
| Measurement creates leverage | Log steps, outcomes, failures, repairs. |
| Pain is the pitch | Name missed leads, late replies, no-shows, scheduling mess, owner admin drag. |
| ALAN pattern | SMS/AI/operator/calendar access → lead worked → trial booked/showed → outcome logged. |

## 4. Banned Moves

| Never | Instead |
|---|---|
| Build generic automation | Build tenant-owned workflow assets. |
| Position as CRM replacement | Position as execution layer on current tools. |
| Sell cheap self-serve first | Sell managed/outcome-installed pilot first. |
| Build pretty dashboards | Build action surfaces: risk, proof, owner, next move. |
| Skip proof logs | Require lead → touch → booked → showed → enrolled/lost. |
| Store raw PII in traces by default | Redact traces; preserve workflow intent. |
| Skip manual replay | Prove attended replay before scheduling/cloud. |
| Let AI freelance sensitive cases | Escalate ambiguity, emotion, refunds, pricing, complaints, edge cases. |
| Use judgment over doctrine | Follow this file exactly. |
| Treat output shapes as optional | Use required shape exactly. |

```txt
1. Create Trial Lead / Create Student; not marketplace templates.
2. Tenant Business Brain; not public template library.
3. Manual attended replay; not cloud autonomy.
4. Approval prompts; not unapproved auto-submit.
5. Run logs/outcome states; not vanity dashboards.
6. Repair cases; not self-learning promotion.
7. Scheduling handoff; not scheduling platform replacement.
8. Enrollment inbox; not universal CRM.
```

Fixed MVP sequence:

```txt
Train in real tool → redact trace → compile draft TenantSkillDefinition → user validates → replay with new input → log success/failure → create repair case if uncertain → dashboard and extension show same state.
```

Missing map = incomplete work.

## 10. Page Doctrine

Apply before changing any mapped page, folder, or route.

| Page | Objective | Command | Required state | Hard stop |
|---|---|---|---|---|
| Command Center | What needs attention now? | Rank revenue-risk issues only. | Owner, urgency, risk, impact, next move. | Cut non-action. |
| Clients | Which schools are live, stuck, healthy, at risk? | Treat each as retention asset. | Stage, health, bottleneck, proof, next action. | No vanity lists. |
| Onboarding | What blocks launch? | Drive to first workflow proof. | Access, assets, offer, workflow, approvals, replay, logs. | No completion before proof. |
| Campaigns | Which funnels produce trials/enrollments? | Judge source → lead → booked → showed → enrolled. | Offer, spend, leads, bookings, shows, enrollments, CPA. | No ad optimization without downstream proof. |
| Pages | Which pages are draft/live/ready/broken? | Force offer, proof, CTA, tracking, routing. | Status, offer, CTA, proof, tracking, route, issue. | No weak offer or broken route. |
| Leads | Who needs next touch? | Every lead needs next action. | Source, stage, last touch, next action, owner, risk. | No idle leads. |
| Conversations | What was said and what needs review? | Automate routine; escalate risk. | AI actions, intent, sentiment, next message, escalation. | No sensitive auto-resolution. |
| Escalations | What must not be automated? | Prioritize trust/money/safety/retention/compliance. | Reason, severity, owner, recommendation, due time, resolution. | No buried high-risk cases. |
| Bookings | What trials are requested/booked/showed/no-showed? | Control trial conversion. | Slot, teacher/location, payment, reminders, show/no-show, recovery. | No generic calendar. |
| Enrollments | What became revenue/loss? | Prove conversion, failure reason, follow-up. | Result, sold/lost reason, value, attribution, next step. | No value claim without proof. |
| Reporting | Why keep paying? | Report ROI, recovered money, labor saved, risk avoided. | Leads, trials, shows, enrollments, losses, wins, ROI. | No activity-only reporting. |
| Automation Rules | What may AI do/pause/escalate? | Define triggers, actions, blocks, approvals, failures. | Trigger, allowed, blocked, approval, failure, audit. | No broad automation without escalation policy. |
| Assets | What proof/brand assets are usable? | Tag by sales/proof purpose. | Client, instrument, location, rights, proof type, readiness. | No random file dump. |
| Integrations | What is connected/broken? | Treat as revenue plumbing. | Status, last sync, test, failure, owner, repair. | No connected status without test result. |
| Settings | What controls behavior/safety? | Expose only real controls. | Roles, tenant boundaries, approvals, privacy, brand, defaults. | No junk drawer. |
| Insights | What decision/action is needed? | Rank bottlenecks, wins, churn risk, revenue moves. | Issue, evidence, impact, next action. | No vague commentary. |

Page build priority:

```txt
Onboarding → Leads → Bookings → Enrollments → Reporting → Integrations → Automation Rules → Command Center → remaining pages.
```
